description: Початок нової задачі в клієнтському проєкті. Збирає контекст, уточнює незрозумілі місця, створює файл задачі.
---

Ти допомагаєш сформулювати задачу для реалізації. Твоя мета — зібрати достатньо контексту щоб план був чітким і однозначним.

## Крок 0: Перевірка проєкту

Переконайся що проєкт ініціалізовано — має бути `.claude/client.md`. Якщо немає — скажи: "Проєкт ще не ініціалізовано. Запусти /client-init спочатку."

## Крок 1: Завантаження контексту

Прочитай:
1. `.claude/client.md` — контекст клієнта
2. `.claude/CLAUDE.md` — правила проєкту та критичні нотатки
3. `.claude/tasks/` — попередні задачі (для розуміння історії)

## Крок 2: Збір контексту задачі

Запитай користувача (якщо він вже не надав у запиті):

1. **Проблема** — що саме не працює або що потрібно зробити? Конкретний опис, не загальний.
2. **Мета** — який результат має бути? Як зрозуміти що задача виконана?
3. **Scope** — що ВХОДИТЬ у задачу, а що точно НЕ входить?
4. **Обмеження** — дедлайни, технічні обмеження, залежності від інших задач?
5. **Контекст із client.md** — як ця задача пов'язана з:
   - Цільовою аудиторією клієнта?
   - Відомими проблемами клієнта?
   - Щоденним флоу користувачів?

## Крок 3: Уточнення (макс. 8 раундів)

Перевір чи контексту достатньо для планування. Якщо є незрозумілі місця — задавай уточнюючі питання.

**Лічильник раундів:** Явно рахуй раунди уточнень (1/8, 2/8, ...). Після 8 раундів — переходь до фіксації assumptions.

**Якщо користувач відповідає "не знаю":**
- Не зупиняйся. Це не кінець уточнення.
- Запропонуй 2-3 конкретні варіанти: "Давай звузимо. Ось три найтиповіших варіанти: (A) ..., (B) ..., (C) ... Який найближчий?"
- Якщо навіть після цього немає відповіді — обери сам на основі контексту клієнта і познач: "(обрав A, бо <причина>. Якщо хочеш змінити — скажи)"

**Критерії достатності контексту:**
- Зрозуміло ЩО робити (проблема і мета)
- Зрозуміло ДЕ робити (які файли/модулі)
- Зрозуміло ЯК перевірити (критерії готовності)
- Немає суперечливих вимог

## Крок 4: Категоризація

Визнач розмір задачі:

| Категорія | Ознаки |
|---|---|
| **tiny** | 1-2 рядки, typo, зміна кольору/тексту, простий конфіг |
| **small** | Один файл, одна функція, простий багфікс, валідація |
| **medium** | 2-5 файлів, нова фіча, рефакторинг модуля |
| **large** | 5+ файлів, архітектурна зміна, міграція, зміна API |

Покажи категорію користувачу і запитай підтвердження.

## Крок 5: Створення файлу задачі

Створи `.claude/tasks/<YYYY-MM-DD>_<slug>.md`:

```markdown
# Task: <коротка назва>
Date: YYYY-MM-DD
Status: in_progress
Category: tiny | small | medium | large

## Problem
<що не працює / що потрібно зробити>

## Goal
<який результат, як перевірити>

## Scope
### In
- <що входить>
### Out
- <що НЕ входить>

## Context from client.md
- **Аудиторія:** <як це впливає на задачу>
- **Проблеми клієнта:** <як задача їх вирішує>
- **Обмеження:** <технічні/доменні обмеження>

## Constraints
- <дедлайни, залежності, технічні обмеження>

## Unresolved questions
<питання на які не вдалося отримати відповідь>

## Assumptions
<що було прийнято як припущення через відсутність відповіді>
```

## Крок 6: Перевірка на нову інформацію

Запитай: "Чи дізнався ти щось нове про клієнта під час обговорення цієї задачі?" Якщо так — онови `.claude/client.md` (додай у відповідну секцію і в "Історію оновлень").

## Крок 7: Фінал

Виведи підсумок:
- Файл задачі: `.claude/tasks/<date>_<slug>.md`
- Категорія: tiny/small/medium/large
- Нез'ясовані питання: N
- Assumptions: N

Для **tiny**: "Можна реалізовувати одразу."
Для **small**: "Якщо готовий — запускай /task-plan для планування."
Для **medium/large**: "Контекст зібрано. Запускай /task-plan для планування. Після плану — обов'язково /preflight."
